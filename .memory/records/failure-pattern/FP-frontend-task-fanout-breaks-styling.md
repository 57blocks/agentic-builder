# task-breakdown — frontend over-fan-out breaks styling cohesion

## What this records
When the frontend task breakdown fans out to **one page per task**, each page is
generated by an independent coding agent that never sees the other pages. Shared
shell/layout, component reuse, spacing scale, and palette application drift apart
across pages — the result looks visually inconsistent and is a poor reproduction
of the design references, even when the DesignSpec is good.

Observed concretely: same PRD + same reference screenshots, two runs.
- Old pipeline grouped the frontend into ~6 multi-page tasks → **cohesive** styling.
- New pipeline (auto-split) produced ~30 one-page tasks (admin alone split into 9)
  → visibly **fragmented** styling, lower screenshot fidelity.

## Why it happens
The original missing-page **404s** came from a *single* task being assigned too
many view files — the agent ran out of attention/budget and silently skipped some
pages. The over-correction (split every multi-view task down to singletons) fixed
404s but pushed the "pages per task" knob to 1, destroying cross-page cohesion.

## How to avoid
- **Do NOT split frontend tasks to one-page-per-task.** Cap pages per task at a
  moderate value (deterministic cap = `MAX_VIEWS_PER_FRONTEND_TASK`, currently 3)
  and keep small related page clusters in a single task so one agent builds them
  with a consistent visual language.
- Split only tasks that *exceed* the cap, into contiguous blocks of ≤cap (in the
  LLM's original file order). Avoid domain-prefix grouping — it collapses back to
  singletons when each page has a distinct prefix (Private/Group/Lecture/Camp…).
- 404-safety does NOT require one-task-per-page: the planning-time
  **page-coverage gate** (`gates/page-coverage-gate.ts` +
  `self-heal/page-coverage-repair.ts`) guarantees every PRD page is covered by
  *some* task — a grouped task counts (matched on coversRequirementIds / title /
  files.creates).
- Prefer an explicit **Frontend Foundation** task (design tokens/theme → shared
  layout shell → shared UI primitives → router skeleton) that every page task
  `depends on` and `reads`, so separate agents reuse one shell instead of each
  re-inventing styling.

## Resolution (2026-06-01)
`splitMultiPageFrontendTasks` changed from "≥2 views → one task per page" to
"≤`MAX_VIEWS_PER_FRONTEND_TASK` kept as-is; larger tasks chunked into contiguous
≤cap blocks". On the reference dataset this took the frontend from 20 one-page
tasks down to 8 cohesive tasks while keeping each agent's page count ≤3.

## DesignSpec.md is a guide DOC, not shared CODE
A common misconception: "DesignSpec.md already guides styling, so pages will be
consistent." They will NOT. A spec doc can only *inform* an agent; it cannot
*enforce* reuse. Each page agent independently transcribes the spec's values into
its own inline markup, so N agents = N divergent transcriptions.

Observed in a real generated frontend (same DesignSpec for all pages):
- Colors hardcoded as arbitrary hex in views, never as tokens — and the SAME
  "primary green" drifted across agents: `bg-[#7d976b]` (×42), `bg-[#6f875f]`
  (×37), `text-[#4c6a3c]` — three different greens for one brand color. Zero
  `var(--)` / `bg-primary` usage in views.
- No shared `Button` primitive at all; `Card` re-invented 13× (MessageCard,
  CourseCard, BillCard, LessonCard, …); `components/ui/` left near-empty.
- index.css `@theme` had typography tokens but NO semantic color tokens.

Why fewer tasks looked better is the same mechanism: fewer frontend tasks = fewer
independent transcriptions = less drift. It is NOT that the good run had a
foundation — neither run did.

## The real fix is code-level enforcement (Frontend Foundation task)
- Synthesize/guarantee a **Frontend Foundation** task that ships actual code:
  semantic design tokens (`--primary` etc. → `@theme` / Tailwind), a layout shell,
  and shared UI primitives (Button/Card/Input/Table/Badge/Modal/EmptyState), plus
  a router skeleton with all routes pre-registered (missing view → build error,
  not silent 404).
- Make every Frontend page task `depends on` it and `reads` its files.
- Add a coding rule: **pages MUST use semantic tokens / shared primitives; inline
  arbitrary hex (`bg-[#...]`) and per-page re-implementations of shared components
  are forbidden.** This makes consistency mechanical regardless of agent count.
- The page-count cap (above) reduces drift; the Foundation eliminates it. They are
  complementary, not alternatives.

## Related
